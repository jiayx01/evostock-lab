# 美股决策持久化记忆策略

## 目标

“越用越聪明”不等于让模型凭印象记住过去，也不等于根据最近一次盈亏自动改策略。目标是把每次决策变成可重放的 episode：当时看到了什么、为什么这样判断、用户后来做了什么、结果如何、这条经验是否经过足够样本验证。下一次分析先召回相似 episode，再用当前事实重新判断。

![图：持久化记忆如何参与下一次判断](assets/evostock-memory-architecture.svg)

*图｜当前事实直接进入决策上下文；历史案例只提供参考，候选经验只有经过时间顺序验证并由用户确认后，才能成为可约束动作的生效规则。*

## 五层记忆

|层级|权威来源|保存内容|能否直接改变建议|
|---|---|---|---|
|事实层|broker generation、行情条、SEC/公司来源、候选事件|当时可见的成交、持仓、价格、市场和公司事实|能，但必须通过完整性门禁|
|决策层|`data/decision_log.jsonl`、完整报告|动作、条件、禁止动作、理由因子、反证、知识截止点和规则版本|是当前决策的审计依据|
|结果层|`data/outcome_price_bars.csv`、`data/decision_outcomes.csv`|实际、原样持有和建议反事实在固定窗口的收益与回撤|不能单独改变下一次建议|
|经验层|`experience/candidate_rules.md`、日周月复盘|错误类型、适用边界、支持证据、反证和待验证规则|否，只能作为待验证上下文|
|规则层|`experience/approved_rules.md`、已确认 skill 版本|通过门禁的生产规则及版本、验证窗口和回滚方式|可以|

自动化 memory 不属于投资知识层。它只保存最近成功 session、已处理 decision、同步水位、未解决异常和报告路径，避免重复工作；不能用一段自然语言 memory 覆盖事件账本或已生效规则。

## 决策 episode

每个 decision 必须绑定同一 `decision_id`，至少包含：

1. `knowledge_cutoff`、市场阶段、持仓来源和数据哈希。
2. 结构化 `market_regime`，包括趋势、广度、信用、波动和行业环境。
3. `reason_factors`。因子使用稳定名称，例如 `market_breadth`、`credit_risk_appetite`、`relative_strength`、`valuation_expectation`、`earnings_risk`、`security_structure_parity`、`thesis_change`。每个因子保留方向、强度、事实值、来源时点和置信度。
4. 主动作、条件、当前禁止动作、最强反证、待确认项和标准化 `recommended_exposure`。
5. 当次使用的提示词、skill、候选策略和生效规则哈希。
6. 实际召回的历史 decision_id 及其成熟结果，防止事后声称“参考过历史”。
7. `episode_id` 与独立样本标记。同 ticker、同 thesis、同主要因子且没有新证据的连续建议属于同一个 episode；每天重复一次不能累计成多个独立信号。

原始 episode 只追加、不回写。后续成交、结果更新、用户明确因果确认和规则评估分别追加新事件。

结果行情只接受完成的一分钟或正式日收盘。盘中 `bar_at` 表示一分钟区间起点，`collected_at` 必须不早于区间结束；旧账本中的进行中分钟只能作为观察，不参与结果。相同经济时点的两个完成价格若不一致，必须失败关闭。建议反事实从 `EMAIL_SENT` 后第一根精确完成分钟开始；缺失时保持 `PENDING_DATA`，不能用送达前分析价回填。

## 用户操作记录

- 已核验 broker events 是操作事实来源；订单提交、撤单或邮件主题不能冒充成交。
- 自动匹配只记录 `建议前已发生 / 建议后成交 / 无成交 / 因果待确认`。
- 只有用户明确表示“这是依据该建议执行”时，才追加 `USER_CAUSALITY_CONFIRMED`。不得从时间接近推断采纳。
- 手续费、成交时间或证券结构不完整时，实际结果保持部分成熟或待确认，不能填零。

## 召回策略

下一次决策按以下顺序检索：

1. 同 ticker 和同证券结构；新 ADR/ADS 比例或公司行动后不与旧结构混用。
2. 同动作与同理由因子，例如“减仓候选 + 估值透支”，而不是只按涨跌相似。
3. 同市场热度、广度、信用和行业环境。
4. 只使用当前 `knowledge_cutoff` 之前已经成熟的结果，优先 5/20 个交易日与最大回撤，同时保留 1 小时和收盘结果的短期含义。
5. 返回相似案例、失败案例、样本数、覆盖率和反证。没有足够样本时明确写“历史样本不足”。

召回结果只是上下文。当前事实优先级更高；未经批准的候选经验不能变成硬规则。

## 学习与晋级

- 每日：补算成熟窗口，记录数据质量、错误类型和一条候选经验，不因单次盈亏调规则。
- 每周：按市场环境、动作、ticker/行业和因子分组，比较实际、原样持有和建议反事实，报告样本数、覆盖率、成本后结果及最差情形。
- 每月：对候选规则做时间顺序 walk-forward。至少 20 个独立信号、多个市场环境、留出期风险调整结果改善且最大回撤不恶化，才生成 skill proposal。
- 生效：用户确认后才更新 `approved_rules` 或生产 skill；保存旧版本、diff、验证命令和回滚方式。

## 防止“越学越偏”

1. 禁止未来数据回填当时事实，所有计算都使用显式 `as_of`。
2. 同时保留成功、失败、暂停和移出样本，避免幸存者偏差。
3. 交易成本、缺失数据和未送达邮件必须进入结果状态。
4. 没有送达后可执行参考价的旧决策不得进入规则晋级样本。
5. 市场环境、证券结构或数据来源变化时分层评估，不能把旧结论机械迁移。
6. 规则升级看风险调整结果和最大回撤，不只看胜率或总收益。
7. 任何自动学习都不能绕过 Gmail 身份、持仓完整性、候选一致性和用户最终决定权。

## 存储与恢复

- `data/` 或独立 `EVOSTOCK_DATA_DIR` 是私有事实源，Git 中只保存代码和公开规则；不得把 Gmail 正文、持仓或成交明细提交到仓库。
- 自动化 memory 只是可丢弃的运行检查点。即使它损坏，也必须能从 broker generation、decision log、price bars、outcomes 和规则版本重建状态。
- 建议把 `EVOSTOCK_DATA_DIR` 放在启用磁盘加密和版本化备份的位置，并定期生成包含文件大小、SHA-256、最后事件 ID 和 schema version 的快照 manifest。备份必须加密，恢复时先校验 manifest，再恢复 active pointer。
- 至少定期做一次只读恢复演练。未验证可恢复的本机单份文件不能称为持久化记忆。

## 实施顺序

1. 先在每日午夜决策中强制写入结构化理由因子、规则哈希和召回记录。
2. 复用现有结果计算器，按最新已完成交易日补齐成熟结果。
3. 再增加确定性的 episode 索引和相似检索器；索引是派生缓存，随时可由事件账本重建。
4. 样本积累后再做分组回测和规则提案，不提前引入黑箱模型或自动调参。
